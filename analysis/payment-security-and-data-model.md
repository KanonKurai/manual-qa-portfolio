# Анализ требований к безопасности платежей и модели данных

## Цель

Собрать список возможных требований к безопасной обработке платежей в интернет-магазине и выделить зоны, которые нужно подтвердить на backend.

Документ подготовлен как пример QA-анализа платежного сценария: статусы заказа, платежные попытки, webhook, идемпотентность, повторные оплаты и связь заказа с платежным провайдером.

## Контекст

Сценарий: одностадийная оплата через внешний платежный сервис.  
Данные банковских карт на стороне интернет-магазина не хранятся.  
Финальный статус оплаты должен определяться на backend по событию от платежного провайдера, а не только по возврату пользователя на сайт.

---

## 1. Что нужно обсудить с командой

- какие требования из списка действительно нужны для текущего этапа;
- что уже реализовано и требует только проверки;
- что нужно дополнительно подтвердить на backend;
- какие пункты можно отложить на следующие этапы;
- какие сценарии нужно покрыть тестами.

Важно: часть пользовательских сценариев можно проверить через UI, DevTools и тестовый платежный кабинет, но реализацию некоторых механизмов должен подтвердить backend.

Например:

- идемпотентность создания платежа;
- обработка повторных webhook-событий;
- защита от параллельного создания платежей;
- корректность смены статусов на backend;
- хранение связи между заказом и платежом.

---

## 2. Контроль статусов заказа

### Требования

- Использовать отдельные внутренние статусы заказа.
- Не смешивать статусы заказа со статусами платежного провайдера.
- Запрещать оплату заказов в финальных статусах.
- Проверять статус заказа на backend перед созданием платежа.

### Пример ожидаемой логики

- `created` — заказ создан.
- `pending_payment` — заказ ожидает оплаты.
- `paid` — заказ оплачен.
- `canceled` — заказ отменен.

### Что важно проверить

- Заказ в статусе `paid` нельзя оплатить повторно.
- Заказ в статусе `canceled` нельзя оплатить.
- Неуспешная оплата не переводит заказ в `paid`.
- Возврат пользователя на сайт сам по себе не меняет заказ на `paid`.

---

## 3. Создание платежа только для валидного заказа

### Требования

- Создавать платеж только для заказа, который может быть оплачен.
- Отклонять создание платежа, если заказ уже оплачен или отменен.
- При несоответствии статуса возвращать ошибку `4xx`.

### Что важно проверить

- Для заказа в статусе `pending_payment` платеж создается.
- Для заказа в статусе `paid` новый платеж не создается.
- Для заказа в статусе `canceled` новый платеж не создается.
- Пользователь видит корректное сообщение, если оплатить заказ уже нельзя.

---

## 4. Webhook как источник истины

### Требования

- Финальный статус оплаты должен обрабатываться через webhook от платежного провайдера.
- Frontend redirect не должен считаться подтверждением успешной оплаты.
- Возврат пользователя на сайт не должен сам по себе переводить заказ в статус `paid`.

### Что важно проверить

- Если пользователь успешно оплатил заказ и не вернулся на сайт, статус заказа все равно обновляется.
- Если пользователь вернулся на сайт без оплаты, заказ не становится оплаченным.
- После обновления страницы статус заказа остается корректным.
- UI отображает результат, соответствующий фактическому статусу заказа.

---

## 5. Идемпотентность операций

### Требования

- Повторный запрос на создание платежа не должен создавать дубли.
- Для одного заказа не должно появляться несколько активных платежей без необходимости.
- Повторная обработка одного и того же события не должна ломать состояние заказа.

### Что важно проверить

- Двойной клик по кнопке “Оплатить” не создает два платежа.
- Одна платежная ссылка не позволяет оплатить заказ повторно.
- Повторный переход по платежной ссылке после оплаты не создает новый платеж.
- Повторная обработка результата оплаты не создает дубли чеков или платежей.

---

## 6. Защита от параллельных действий

### Требования

- Исключить создание нескольких платежей для одного заказа при быстрых или параллельных действиях пользователя.
- Сохранять корректные статусы заказа и платежа при повторных действиях.

### Пользовательские сценарии для проверки

- Двойной клик по кнопке “Оплатить”.
- Открытие одной платежной ссылки в двух вкладках.
- Попытка оплатить один заказ с двух открытых страниц платежного сервиса.
- Повторная оплата через другую сессию или инкогнито.
- Сценарий “назад-вперед” в браузере.
- Обновление платежной страницы после запуска оплаты.

---

## 7. Контроль повторных оплат

### Требования

- Нельзя повторно оплатить уже оплаченный заказ.
- Перед каждой попыткой оплаты backend должен проверять текущий статус заказа.
- Оплаченный заказ должен быть защищен от повторного создания платежа.

### Что важно проверить

- После успешной оплаты повторная оплата недоступна.
- Повторный переход по старой платежной ссылке не позволяет оплатить заказ второй раз.
- После успешной оплаты обновление страницы не создает новый платеж.

---

## 8. Ограничение попыток оплаты

### Возможные требования

- Лимитировать количество попыток оплаты на один заказ.
- Ограничивать слишком частые попытки оплаты.
- Обрабатывать подозрительную активность.

### Что нужно обсудить

- Нужны ли лимиты на текущем этапе.
- Где должен применяться лимит: на заказ, пользователя, IP или платежную попытку.
- Какое сообщение должен видеть пользователь при превышении лимита.

---

## 9. 3-D Secure

### Требования

- Поддерживать прохождение 3-D Secure, если оно требуется платежным провайдером.
- Корректно обрабатывать успешное и неуспешное прохождение 3-D Secure.

### Что важно проверить

- Открывается страница подтверждения 3-D Secure.
- После успешного прохождения 3-D Secure заказ становится оплаченным.
- При ошибке 3-D Secure заказ не получает статус `paid`.
- Повторное прохождение 3-D Secure по той же странице не создает второй платеж.

---

## 10. Логирование и аудит

### Что желательно логировать

- создание заказа;
- создание платежа;
- платежные попытки;
- webhook-события;
- ошибки создания платежа;
- ошибки обработки webhook;
- причины отмены платежа;
- связь заказа и платежа.

### Зачем это нужно

Логи помогают восстановить цепочку событий:

1. заказ создан;
2. платеж создан;
3. пользователь перешел к оплате;
4. платеж получил финальный статус;
5. заказ обновил статус;
6. чек сформирован или не сформирован.

---

## 11. Обработка ошибок платежного провайдера

### Требования

- Корректно обрабатывать ошибки API платежного сервиса.
- Не показывать пользователю ошибку пользовательских данных, если произошла серверная ошибка.
- Сохранять причину неуспешной оплаты, если она доступна.

### Что важно проверить

- При ошибке создания платежа пользователь не попадает на платежную страницу.
- При ошибке оплаты заказ не становится оплаченным.
- Причина отказа сохраняется и доступна для анализа.
- UI показывает корректное сообщение об ошибке.

---

## 12. Авто-отмена неоплаченных заказов

### Возможные требования

- Ввести время жизни заказа в статусе ожидания оплаты.
- Автоматически переводить просроченные заказы в `canceled`.
- Запрещать оплату после авто-отмены.

### Что важно проверить

- После истечения времени заказ переходит в `canceled`.
- Просроченную платежную ссылку нельзя использовать для оплаты.
- Повторный переход по старой ссылке не создает новый платеж.
- UI показывает понятный результат для просроченной оплаты.

---

# Модель данных для платежного сценария

## 1. Данные заказа

### Минимально нужны

- `order_id` — внутренний ID заказа;
- `order_number` — номер заказа;
- `order_status` — внутренний статус заказа;
- `amount_total` — итоговая сумма заказа;
- `currency`;
- `created_at`;
- `updated_at`.

### Желательно хранить

- выбранный способ оплаты;
- email покупателя;
- телефон покупателя;
- причину отмены заказа;
- snapshot состава заказа на момент оформления;
- данные доставки или получателя.

### Важно

Статусы заказа должны храниться отдельно от статусов платежного провайдера.

---

## 2. Данные платежной попытки

### Минимально нужны

- внутренний ID платежной попытки;
- `order_id` — связь с заказом;
- платежный провайдер;
- ID платежа у провайдера;
- последний известный статус платежа;
- причина отмены платежа, если есть;
- сумма;
- валюта;
- idempotency key;
- дата создания;
- дата обновления;
- время последнего webhook.

### Желательно хранить

- дату создания платежа у провайдера;
- ссылку на подтверждение платежа;
- ключевые поля ответа при создании платежа;
- признак финального статуса;
- техническую ошибку создания платежа, если она была.

### Важно

По одному заказу может быть несколько платежных попыток, поэтому лучше хранить не только текущий платеж, но и историю попыток.

---

## 3. Данные от платежного провайдера

### Минимально нужны

- ID платежа;
- статус платежа;
- признак успешной оплаты;
- сумма;
- валюта;
- metadata;
- причина отмены, если платеж отменен.

### Желательно хранить

- дату создания платежа;
- тип подтверждения;
- ссылку на подтверждение;
- описание платежа;
- тип платежного метода.

### Важно

Статус платежа и причина отмены — разные сущности.

Например:

- `pending`, `succeeded`, `canceled` — это статусы;
- `expired`, `3d_secure_failed` и похожие значения — это причины отмены или отказа.

---

## 4. Связь заказа и платежа

### Нужная связка

- внутренний заказ должен быть связан с ID платежа у провайдера;
- внутренний `order_id` должен передаваться в metadata платежа.

### Желательно

- передавать в metadata номер заказа;
- передавать окружение, если это помогает диагностике.

### Важно

Связь только по описанию платежа или только по номеру заказа недостаточно надежна. Нужна однозначная техническая связь между заказом и платежом.

---

## 5. Данные webhook

### Минимально нужны

- тип события;
- ID платежа у провайдера;
- raw body события;
- время получения;
- статус обработки.

### Желательно хранить

- время обработки;
- найденный `order_id`;
- найденную платежную попытку;
- текст ошибки обработки;
- заголовки запроса.

### Важно

Webhook должен быть источником истины по финальному статусу платежа.

---

# Обязательные правила работы

## Минимально нужно

- Финальный статус оплаты определяется по webhook.
- Возврат пользователя на сайт не подтверждает оплату.
- Создание платежа должно быть идемпотентным.
- Для оплаченного заказа нельзя создавать новый платеж.
- Причина отмены платежа должна сохраняться отдельно.
- Между заказом и платежом должна быть однозначная связь.

## Желательно

- Хранить историю всех платежных попыток по заказу.
- Хранить журнал webhook отдельно.
- Логировать результат обработки webhook.
- Хранить время последнего webhook по платежу.

---

# Что не нужно хранить

- номер карты;
- CVV/CVC;
- срок действия карты;
- любые чувствительные платежные данные.

---

# Короткий вывод

Минимально нужны три блока данных:

- заказ;
- платеж / платежная попытка;
- webhook-события.

Основная связка:

- внутренний ID заказа;
- ID платежа у провайдера;
- ID заказа в metadata платежа.

Такой подход помогает избежать повторных оплат, потери связи между заказом и платежом, некорректных статусов и ситуаций, когда пользователь видит успешный результат без успешной оплаты.
