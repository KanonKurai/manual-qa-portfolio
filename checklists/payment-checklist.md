# Чек-лист тестирования оплаты

## Цель

Проверить полный платежный сценарий в интернет-магазине: создание заказа, переход к оплате, успешную и неуспешную оплату, отмену пользователем, работу статусов, защиту от дублей, чеки и возврат пользователя на сайт.

## Область проверки

- создание заказа;
- переход к платежному сервису;
- оплата банковской картой;
- 3-D Secure;
- успешные и неуспешные платежи;
- отмена оплаты пользователем;
- таймаут платежной страницы;
- повторные действия пользователя;
- статусы заказа и платежа;
- чеки;
- webhook-обработка;
- возврат пользователя на сайт;
- согласованность данных между сайтом, backend и платежным сервисом.

---

## 1. Успешная оплата картой

- [ ] Кнопка “Оплатить” активна при валидных данных.
- [ ] По кнопке “Оплатить” происходит переход на страницу платежного сервиса.
- [ ] Сумма заказа соответствует сумме платежа.
- [ ] Стоимость доставки учитывается в итоговой сумме оплаты.
- [ ] Номер заказа на сайте соответствует номеру заказа в платежном сервисе.
- [ ] Валюта платежа отображается корректно.
- [ ] На стороне платежного сервиса отображается идентификатор платежа.
- [ ] С валидной картой оплата проходит успешно.
- [ ] После успешной оплаты доступен возврат на сайт.
- [ ] После возврата на сайт заказ имеет статус “Оплачен”.
- [ ] Статус заказа обновляется даже без возврата пользователя на сайт.
- [ ] Статус заказа не меняется некорректно после перезагрузки страницы.
- [ ] Заказанные товары соответствуют оплаченным товарам в рамках одного заказа.
- [ ] После успешной оплаты не создаются дубли заказа.
- [ ] После успешной оплаты не создаются дубли платежа.
- [ ] Двойной клик по кнопке “Оплатить” не создает два заказа или два платежа.
- [ ] При медленной сети платеж проходит корректно.
- [ ] Оплата одновременно из двух устройств с одного аккаунта создает разные заказы и разные платежи.
- [ ] Оплата одного заказа из двух открытых страниц платежного сервиса не создает второй успешный платеж.
- [ ] Оплаченный заказ нельзя оплатить повторно по той же платежной ссылке.
- [ ] Оплаченный заказ нельзя оплатить повторно по той же платежной ссылке в другой сессии или инкогнито.
- [ ] Оплаченный заказ нельзя оплатить повторно через сценарий “назад-вперед” в браузере.
- [ ] При обновлении страницы платежного сервиса после запуска оплаты нельзя совершить повторный платеж.

---

## 2. Успешная оплата с 3-D Secure

- [ ] Открывается страница 3-D Secure, если карта требует подтверждения.
- [ ] 3-D Secure можно успешно пройти.
- [ ] После успешного прохождения 3-D Secure оплата проходит успешно.
- [ ] После успешного прохождения 3-D Secure заказ получает статус “Оплачен”.
- [ ] Прохождение 3-D Secure с двух открытых страниц не создает второй успешный платеж.
- [ ] После успешного прохождения 3-D Secure нельзя совершить повторный платеж по той же странице подтверждения.
- [ ] После успешного прохождения 3-D Secure нельзя совершить повторный платеж через сценарий “назад-вперед”.
- [ ] При обновлении страницы 3-D Secure после запуска подтверждения нельзя совершить повторный платеж.

---

## 3. Отмена оплаты пользователем и незавершенная оплата

- [ ] Незавершенная оплата не создает успешный платеж.
- [ ] Закрытие страницы платежного сервиса без оплаты не переводит заказ в статус “Оплачен”.
- [ ] Возврат на сайт без оплаты не показывает пользователю успешное оформление заказа.
- [ ] При незавершенной оплате заказ отображается в корректном статусе.
- [ ] Статус заказа не меняется на “Оплачен” после перезагрузки страницы.
- [ ] Через заданное время неоплаченный заказ переходит в статус “Отменен”, если такая логика предусмотрена требованиями.
- [ ] Незавершенная оплата одного заказа из двух открытых страниц платежного сервиса не создает два успешных платежа.
- [ ] При незавершенной оплате заказ можно оплатить по той же платежной ссылке, если ссылка еще активна.
- [ ] При незавершенной оплате заказ можно оплатить по той же платежной ссылке в другой сессии или инкогнито, если ссылка еще активна.
- [ ] При незавершенной оплате заказ можно оплатить через сценарий “назад-вперед”, если ссылка еще активна.
- [ ] После истечения времени жизни платежной ссылки страница платежного сервиса становится просроченной.
- [ ] После истечения времени жизни платежной ссылки заказ нельзя оплатить по старой ссылке.
- [ ] После истечения времени жизни платежной ссылки повторный переход по старому URL не позволяет выполнить оплату.

---

## 4. Неуспешная оплата картой

- [ ] При ошибке 3-D Secure оплата не проходит.
- [ ] При отказе банка оплата не проходит.
- [ ] При истекшем сроке действия карты оплата не проходит.
- [ ] При подозрении на мошенничество по карте оплата не проходит.
- [ ] При недостаточном количестве средств на карте оплата не проходит.
- [ ] При неправильном номере карты оплата не проходит.
- [ ] При неправильном CVV/CVC оплата не проходит.
- [ ] При недоступности банка оплата не проходит.
- [ ] При исчерпанном лимите по карте оплата не проходит.
- [ ] При запрете на использование карты оплата не проходит.
- [ ] При ошибке без детальной причины оплата не проходит.
- [ ] Неуспешная оплата не создает успешный платеж.
- [ ] При неуспешной оплате заказ не получает статус “Оплачен”.
- [ ] При неуспешной оплате заказ отображается в корректном статусе.
- [ ] Статус заказа не меняется на “Оплачен” после перезагрузки страницы.
- [ ] Через заданное время неоплаченный заказ переходит в статус “Отменен”, если такая логика предусмотрена требованиями.
- [ ] После истечения времени жизни платежной ссылки страница платежного сервиса становится просроченной.
- [ ] После истечения времени жизни платежной ссылки заказ нельзя оплатить по старой ссылке.
- [ ] Несколько неуспешных попыток оплаты не создают успешный платеж.
- [ ] Несколько неуспешных попыток оплаты не создают дубли заказа.
- [ ] Причина неуспешной оплаты корректно сохраняется и доступна для анализа.

---

## 5. Повторные действия и защита от дублей

- [ ] Повторный клик по кнопке “Оплатить” не создает несколько платежей для одного заказа.
- [ ] Повторный клик по кнопке “Оплатить” не создает несколько одинаковых заказов.
- [ ] Повторное обновление страницы после оплаты не создает новый платеж.
- [ ] Повторный переход по платежной ссылке после успешной оплаты не позволяет оплатить заказ повторно.
- [ ] Повторный переход по платежной ссылке после неуспешной оплаты обрабатывается корректно.
- [ ] Переход по платежной ссылке из другой сессии обрабатывается корректно.
- [ ] Переход по платежной ссылке из режима инкогнито обрабатывается корректно.
- [ ] Сценарий “назад-вперед” в браузере не приводит к повторной оплате.
- [ ] Открытие одной платежной ссылки в двух вкладках не создает два успешных платежа.
- [ ] При одновременных действиях пользователя статусы заказа и платежа остаются согласованными.

---

## 6. Webhook-обработка

- [ ] После успешной оплаты статус заказа обновляется без ручного обновления страницы пользователем.
- [ ] После успешной оплаты статус заказа обновляется, даже если пользователь не вернулся на сайт.
- [ ] После успешной оплаты статус заказа обновляется при обновлении страницы через некоторое время.
- [ ] Frontend redirect не является единственным основанием для смены статуса заказа на “Оплачен”.
- [ ] Финальный статус оплаты определяется на стороне backend.
- [ ] Повторная доставка webhook-события не ломает статус заказа.
- [ ] Повторная доставка webhook-события не создает дубли платежей.
- [ ] Повторная доставка webhook-события не создает дубли чеков.
- [ ] При ошибке обработки webhook система может корректно обработать повторное событие.
- [ ] Статус заказа, статус платежа и отображение в UI остаются согласованными.

---

## 7. Чеки

- [ ] Чек формируется после успешной оплаты.
- [ ] Сумма чека соответствует сумме заказа.
- [ ] Состав товаров в чеке соответствует составу заказа.
- [ ] Количество товаров в чеке соответствует заказу.
- [ ] Цена каждой позиции в чеке соответствует данным заказа.
- [ ] Наименование товаров в чеке передается корректно.
- [ ] Скидка в чеке отображается корректно, если она применена.
- [ ] Стоимость доставки в чеке отображается корректно, если доставка платная.
- [ ] Для заказа с одним товаром чек формируется корректно.
- [ ] Для заказа с несколькими товарами чек формируется корректно.
- [ ] Для заказа с разными типами товаров чек формируется корректно.
- [ ] При неуспешной оплате чек не формируется.
- [ ] При незавершенной оплате чек не формируется.
- [ ] При повторной обработке успешного платежа не создается дубль чека.

---

## 8. Заказ после оформления

- [ ] Заказ создается при нажатии на кнопку “Оплатить”
- [ ] Заказу присваивается уникальный номер.
- [ ] Заказ отображается в списке заказов пользователя.
- [ ] Заказ получает корректный стартовый статус.
- [ ] Статус заказа не меняется некорректно после обновления страницы.
- [ ] Статус заказа меняется в соответствии с жизненным циклом.
- [ ] После успешной оплаты заказ получает статус “Оплачен”.
- [ ] После неуспешной оплаты заказ не получает статус “Оплачен”.
- [ ] После незавершенной оплаты заказ не получает статус “Оплачен”.
- [ ] Не создаются дубли заказа.
- [ ] Один заказ не разбивается на несколько заказов без причины.
- [ ] После создания заказа корзина очищается, если такая логика предусмотрена.
- [ ] Состав заказа сохраняется корректно после перехода к оплате.
- [ ] Состав заказа не меняется после успешной оплаты.
- [ ] Сумма заказа не меняется после успешной оплаты.

---

## 9. Страница возврата на сайт

- [ ] Возврат на сайт со страницы платежного сервиса доступен.
- [ ] После возврата при успешной оплате отображается страница успешного оформления заказа.
- [ ] После возврата при неуспешной оплате не отображается страница успешного оформления заказа.
- [ ] После возврата при незавершенной оплате не отображается страница успешного оформления заказа.
- [ ] После возврата при отмене оплаты пользователем отображается корректный результат.
- [ ] После возврата пользователь понимает, завершена оплата или нет.
- [ ] Отображаемый результат соответствует фактическому статусу заказа на backend.
- [ ] Обновление страницы возврата не меняет статус заказа некорректно.
- [ ] Повторный переход на страницу возврата не создает новый платеж.
- [ ] Повторный переход на страницу возврата не меняет статус заказа без фактической оплаты.

---

## 10. Проверки в платежном сервисе и логах

- [ ] Успешные платежи сохраняются в платежном сервисе.
- [ ] Неуспешные платежи сохраняются с причиной отказа.
- [ ] Номер заказа на сайте соответствует данным в платежном сервисе.
- [ ] Сумма платежа в платежном сервисе соответствует сумме заказа.
- [ ] Валюта платежа в платежном сервисе корректная.
- [ ] Идентификатор платежа связан с заказом.
- [ ] Логи по платежам доступны в течение ожидаемого срока хранения.
- [ ] По логам можно восстановить основную цепочку событий: создание заказа, создание платежа, результат оплаты, обновление статуса.

## 11. Проверки через DevTools / API

- [ ] Проверить запрос создания заказа.
- [ ] Проверить запрос создания платежа.
- [ ] Проверить payload ключевых запросов.
- [ ] Проверить response ключевых запросов.
- [ ] Проверить HTTP-статусы ответов.
- [ ] Проверить статус заказа после успешной оплаты.
- [ ] Проверить статус заказа после неуспешной оплаты.
- [ ] Проверить статус заказа после отмены оплаты пользователем.
- [ ] Проверить статус заказа после перезагрузки страницы.
- [ ] Проверить, что данные в UI соответствуют ответу backend.
- [ ] Проверить отсутствие дублей платежных попыток.
- [ ] Проверить отсутствие дублей заказов.
- [ ] Проверить корректность обработки ошибок backend.
- [ ] Проверить поведение при медленной сети.

---

## 12. Основные риски

- Пользователь может увидеть успешное оформление заказа без успешной оплаты.
- Заказ может получить статус “Оплачен” без подтвержденного платежа.
- Оплаченный заказ может быть доступен для повторной оплаты.
- Для одного заказа может создаться несколько платежей.
- Для одного платежа может создаться несколько чеков.
- Сумма платежа может не совпасть с суммой заказа.
- Доставка или скидка могут некорректно попасть в итоговую сумму.
- Состав заказа может отличаться от состава оплаченных товаров.
- UI, backend и платежный сервис могут показывать разные статусы.
- Статус заказа может зависеть только от возврата пользователя на сайт, а не от backend/webhook.
